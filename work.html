📷 DualCamera

前后双摄同时录制 · 接近原生手感 · 无水印 · 无订阅

iOS 17+ SwiftUI + AVFoundation AVCaptureMultiCamSession HEVC / H.264 最高 4K · 60fps 5 种语言

1快速上手

  • 首次启动：允许 相机 和 麦克风 权限（保存时还会请求一次 相册 写入权限）
  • 在「选择您的相机」页面按顺序点选 2 个镜头——① 是主画面，② 是副画面（画中画的小窗）。不能同时工作的镜头会自动变灰
  • 点 开始 进入拍摄界面，按红色大按钮开始/停止录像，白色 📷 按钮拍照
  • 拍完的视频和照片自动存入系统 照片 App，点左下角缩略图直接跳转查看

2三种拍摄模式

⬛ 分割

两路画面拼在同一画面里：竖屏上下分、横屏左右分。输出 1 个合成文件。

🔳 画中画

主画面全屏，副画面悬浮小窗（默认模式）。输出 1 个合成文件。

⏹⏹ 双录

两个摄像头各自独立录制。输出 2 个独立文件（-back / -front），方便后期分别剪辑。

📐 竖横同拍

在「•••」面板开启后，分割/画中画模式下一次录制同时输出竖版 (9:16) + 横版 (16:9) 两个文件——竖版发抖音，横版发 B 站，不用拍两遍。

怎么切换：底部模式栏点「分割 / 画中画 / 双录」；预览左上角的提示条会显示当前输出是 1 个还是 2 个文件。

4手势操作

  • 单击画面 —— 出现黄色对焦框：点哪对哪（主画面/小窗都可以）；框旁边的 ☀️ 滑轨上下拖动调节曝光亮度
  • 双击画面 —— 前后画面互换
  • 双指捏合 —— 连续变焦（后置主镜头，最大 10×），落在档位之间时变焦条显示实时倍数如 1.7×，录制中也能用

6可靠性保护（自动，无需操作）

📞 来电保护

录制中来电话：自动停止并完整保存已录内容，不会损坏文件。

🌡 过热保护

温度偏高时顶部显示警告；达到临界温度自动停录保片。

⏸ 占用提示

相机被其他 App 抢占时顶部横幅提示，恢复后自动继续预览。

🌗 曝光预热

刚启动/切镜头时短暂显示「对焦中…」并锁定快门，避免录进一段黑屏渐亮的画面。

🎵 不打扰音乐

打开 App 不会掐断正在播放的音乐/播客。

📱 平放防翻转

手机平放（俯拍/架拍）时画面方向锁定，不会因轻微晃动而翻转。

7横屏使用

  • 界面自动重排：预览居左，控制栏移到右侧竖排
  • 分割模式自动变为左右分屏
  • 录制方向在按下录制键那一刻锁定：横着拿录 1920×1080，竖着拿录 1080×1920；录制中转动手机不影响当前片段

8多语言

设置 → 语言，手动选择：English（默认）/ Deutsch / 中文 / 日本語 / Français。不跟随系统语言，选完立即生效并记住。

9已知限制

  • 需要真机且支持 MultiCam（iPhone XS/XR 及之后机型），模拟器无法运行双摄
  • 录制中跨物理镜头切换（如 1× → .5×）会有一瞬间画面停顿（同镜头内数码变焦无此问题）；系统相机的无缝切换依赖 MultiCam 用不了的虚拟镜头
  • 微距为近似实现：选 .5× 超广角时自动启用近距离对焦优化，非自动切换
  • 4K + 60fps + 竖横同拍同时开启负载很高，低端机型可能掉帧发热，建议按需组合

2整体数据流

后置镜头 ──┐ ┌─→ backPreviewLayer（预览，不防抖） ├─ AVCaptureMultiCamSession ──连接──┤ 前置镜头 ──┤ （手动无连接模式组装） ├─→ backVideoOutput ─┐ │ ├─→ frontPreviewLayer │ 麦克风 ──┘ ├─→ frontVideoOutput ─┼─→ AVCaptureDataOutput └─→ audioOutput ─┘ Synchronizer │ (dataQueue) ┌───────────────────────────────────────────┘ ▼ 同步回调：back + front + audio 三路时间对齐的样本 │ ┌───────────────┼──────────────────┐ ▼ ▼ ▼ 拍照触发时 DualStreamRecorder (丢弃，未录制时) CoreImage 合成 │ → JPEG → 相册 ├─ 分割/画中画: CoreImage 合成 → AVAssetWriter ×1 │ (竖横同拍时: 双画布 ×2) └─ 双录: 两路原始帧直写 → AVAssetWriter ×2 │ ▼ .mov 文件 → 系统相册 + 应用沙盒

3核心类详解

DualCameraController —— 引擎总控

持有 AVCaptureMultiCamSession 和两个预览层，是所有相机操作的唯一入口。ObservableObject，约 30 个 @Published 属性驱动 UI。职责：

  • 镜头发现与配对：DiscoverySession 枚举物理镜头（刻意排除 TrueDepth——它与前置广角是同一颗传感器，双选会崩），用 supportedMultiCamDeviceSets 校验两镜头能否同时运行
  • 会话组装：全程「无连接模式」（addInputWithNoConnections / addOutputWithNoConnections + 手动 AVCaptureConnection）——MultiCam 下必须手动接线才能精确控制每路的方向、镜像、防抖
  • 格式协商：applyBestFormat(to:quality:fps:) 在 isMultiCamSupported 的格式里按 分辨率优先→帧率次之 降级选择，然后锁定恒定帧率（min=max FrameDuration）；达不到的档位通过 fourKAvailable/fpsAchieved 上报 UI 显示 ⚠️
  • 镜头切换：同镜头变焦 = 改 videoZoomFactor（档位点击用 ramp() 平滑）；跨镜头 = 只拆装后置那半套输入/连接，前置和录制不动
  • 方向系统：每镜头一个 RotationCoordinator + KVO，预览实时转正；手机平放时（UIDevice.orientation 非有效界面方向）忽略更新防止翻转；录制方向在开录瞬间从预览连接读取并锁定
  • 生命周期防护：五组 NotificationCenter 观察者（会话中断/中断结束/运行时错误/音频中断/热状态），任何一种都能安全收尾正在写的文件

DualStreamRecorder —— 录制器

纯 AVAssetWriter 封装，所有调用都在同一条串行 dataQueue 上，因此内部无锁。三种输出拓扑：

  • 合成单文件（分割/画中画）：AVAssetWriterInputPixelBufferAdaptor + 像素缓冲池，每帧经 DualFrameCompositor 画到画布再编码
  • 合成双文件（竖横同拍）：同一份输入帧渲染到两个不同尺寸画布（9:16 + 16:9），两个 writer 并行写
  • 双录：跳过合成，两路 CMSampleBuffer 直接 append，零转码开销

编码参数显式给定：码率 = 宽×高×fps×每像素比特数（HEVC 0.11 / H.264 0.17，有封顶），关键帧间隔 2 秒；音频 AAC 48kHz 立体声 128kbps。front 帧比 back 晚到时用 latestFront 缓存兜底配对。

DualFrameCompositor —— 合成数学

无状态 enum，纯 CoreImage：aspect-fill 缩放 + 平移 + 裁剪 + composited(over:) 叠加。录像和拍照共用这一份代码，保证视频帧和照片的排版像素级一致。画中画小窗位置/分割线方向都由画布尺寸和 portrait 参数推导。

DualPreviewView —— 预览宿主 + 手势中心

UIViewRepresentable 包一个 UIKit view，直接把 controller 的两个 AVCaptureVideoPreviewLayer 挂为 sublayer——切换模式/交换画面只是改 layer frame，不触碰会话，所以瞬时完成。手势全部在 UIKit 层处理（单击对焦要 captureDevicePointConverted 坐标变换、双击要与单击 require(toFail:) 仲裁、捏合要连续回调），比 SwiftUI 手势更可控。

LocalizationManager —— 手动多语言

单例 + @Published language，默认英文、UserDefaults 持久化、刻意不读系统 locale。文案 = LocKey 枚举 × Strings.table 字典查表，带 {0}/{1} 占位符替换。加语言 = 枚举加一个 case + 表里补翻译，编译器会强制补全。

4线程模型

队列职责规则
sessionQueue（串行）会话配置、镜头切换、变焦、对焦、手电筒所有 beginConfiguration/commitConfiguration 必须在此，避免与 startRunning 竞争
dataQueue（串行）同步器回调、录制器全部读写、拍照触发标志录制器状态只在此队列碰，故无锁；合成渲染也在此（CoreImage 到 GPU）
主队列全部 @Published 更新、预览层方向角、通知回调跨队列传值先拷贝为值类型再 dispatch，满足 Swift 6 Sendable 检查
为什么：AVFoundation 的会话操作是阻塞且非线程安全的；把「配置」「数据」「UI」三类工作分到三条队列，配置卡顿不掉帧、数据处理不卡 UI。类标记 @unchecked Sendable，安全性靠上述队列纪律人工保证。

5关键工程决策

问题方案
同一颗前置传感器被选两次 → MultiCam 崩溃发现列表排除 TrueDepth；选择页用 supportedMultiCamDeviceSets 把不可配对镜头置灰；配置前再校验一次兜底
手机平放时画面随机 180° 翻转重力在水平姿态无法区分正/倒，此时忽略 RotationCoordinator 更新，保持最后有效方向
开录瞬间录进黑屏渐亮KVO 监听两镜头 isAdjustingExposure，曝光收敛前禁用快门（1.5s 超时兜底）
来电/抢占导致文件损坏音频中断和会话中断通知里先 stopRecording() 走完整的 finishWriting 流程再放行
打开 App 掐断用户音乐automaticallyConfiguresApplicationAudioSession = false + 自管 .playAndRecord + .mixWithOthers
4K60 双摄超出机型能力格式协商分辨率优先、帧率降级，结果通过 ⚠️ 角标如实上报，绝不静默谎报
录像和拍照排版不一致合成数学抽成无状态 DualFrameCompositor，两条路径调同一函数
拍照不打断录像不用 AVCapturePhotoOutput（MultiCam 下加输出会打断会话），改为在同步器回调里截取当前帧标志位触发

6构建与调试

  • 环境：Xcode 16+ / Swift 6（严格并发检查开启）/ 部署目标 iOS 17.0
  • 构建：open DualCamera.xcodeproj → 选 DualCamera scheme → 签名页设置自己的 Team → 真机运行（模拟器只能看 UI，MultiCam 不工作）
  • 命令行部署：xcodebuild -project DualCamera.xcodeproj -scheme DualCamera -destination 'id=<设备ID>' -allowProvisioningUpdates build，再用 xcrun devicectl device install app / process launch 安装启动
  • 调试工具：设置 → 开发者控制台，可扫描当前机型全部镜头/格式/帧率/MultiCam 组合并导出 JSON——适配新机型时先跑这个
  • 测试：DualCameraTests 覆盖模型与能力报告导出；相机行为只能真机验证
